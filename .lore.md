## Long-term Knowledge

### Architecture

<!-- lore:019f050e-f7c7-79be-8b33-c51503d1c8b6 -->
* **DynVision documentation uses Diátaxis framework**: DynVision documentation uses the Diátaxis framework (https://diataxis.fr/) with 4 sections: Tutorials, How-to Guides, Reference, Explanation. Hosted on GitHub Pages at https://lindsay-lab.github.io/DynVision/ via MkDocs Material theme. Per-quadrant badges injected via overrides/main.html. Font: Inter throughout (Merriweather override for Explanation was removed). Home page: no nav tab — linked via logo only; no redundant h1; Documentation Structure uses h3 headers per quadrant with bullet sublists. Reference index: bullet lists require blank lines before them (Python-Markdown requirement) or they render flat. Tags plugin removed — it caused duplicate visible badges alongside Diátaxis badges on pages with \`tags:\` front matter.

<!-- lore:019f051e-76fb-7ee4-a6cc-f5b63e76e33f -->
* **DynVision MkDocs config: location, plugins, and deploy**: mkdocs.yml lives inside docs/ with docs\_dir: . (self-referential) and site\_dir: ../site. Theme: Material with custom\_dir=overrides. Logo: assets/logo.svg, favicon: assets/thumbnail.png. Fonts: Inter (text), Fira Code (code) — Merriweather was previously used for Explanation pages but removed so all sections use Inter. Plugins: search, autorefs, mkdocstrings (numpy style), git-revision-date-localized (fallback\_to\_build\_date: true). Version provider: mike — must be in pyproject.toml \[doc] extra. Nav uses navigation.tabs + navigation.tabs.sticky; Home tab removed (home linked via logo only). Deploy: docs.yml uses peaceiris/actions-gh-pages@v3 to gh-pages on push to main. Copyright footer contains preprint DOI link.

<!-- lore:019f050e-f785-78bd-810b-3ac196a1dd00 -->
* **DynVision repo: fork vs upstream remotes**: Two remotes: origin = git@github.com:rgutzen/DynVision.git (Robin Gutzen's fork), upstream = git@github.com:Lindsay-Lab/DynVision.git (org repo). PRs are developed on origin branches and merged to upstream/main. Zenodo integration is connected to origin (rgutzen/DynVision) because Zenodo account only has access to the fork, not the Lindsay-Lab org repo.

### Gotcha

<!-- lore:019f051e-7707-701a-a16f-409d7f6b3724 -->
* **DynVision docs: --strict CI fails due to pre-existing broken links in dev docs**: Trap: adding \`--strict\` to \`mkdocs build\` looks like a good CI quality gate. Fix: DynVision has ~40+ pre-existing broken links concentrated in dev/planning docs (claude-guide.md, documentation-style.md, model-integration.md) and aspirational TOC entries in explanation/index.md (~25 non-existent pages). \`--strict\` promotes ALL warnings to errors regardless of \`validation:\` config, so CI fails on unrelated legacy issues. Chosen approach: fix user-facing navigable broken links, add \`exclude\_docs\` and \`validation: warn\` config, and drop \`--strict\` from the workflow to keep CI meaningful without constant friction.

<!-- lore:019f051e-7712-7ccb-ac60-1a9cd48be8a5 -->
* **DynVision docs: common broken-link traps in existing files**: Several recurring broken-link patterns in DynVision docs: (1) \`tutorial/\` vs \`tutorials/\` — reference/index.md used \`../tutorials/index.md\` but directory is \`tutorial/\`; (2) \`temporal\_dynamics.md\` uses underscore but index.md linked \`temporal-dynamics.md\` (hyphen); (3) \`user-guide/evaluation.md\` doesn't exist — correct page is \`user-guide/model-testing.md\`; (4) dev docs use spurious \`docs/\` prefix in links (e.g. \`docs/explanation/...\` resolving relative to \`development/guides/\`). Always verify actual filenames including separators before linking.

<!-- lore:019f051e-771b-7925-b37f-6c789c35d857 -->
* **DynVision docs: mike missing from pyproject.toml \[doc] extra**: Trap: mkdocs.yml declares \`version.provider: mike\` so it looks configured. Fix: \`mike\` must also be listed in pyproject.toml \`\[doc]\` extra — it was absent, meaning CI installs via \`pip install -e '.\[doc]'\` would fail to find mike and the version selector would break. Added \`mike\` to the \[doc] extra. Also note: \`pillow\` was commented out in \[doc] but is installed post-install in .readthedocs.yml — keep that split intentional.

<!-- lore:019f142a-ee66-70e1-9785-f94584f3fe08 -->
* **DynVision docs: MkDocs tags plugin conflicts with Diátaxis quadrant badges**: Trap: adding \`tags:\` front matter (e.g. \`tags: explanation\`) looks like a harmless metadata annotation. Fix: MkDocs tags plugin renders tags as visible badges in the page, appearing alongside the Diátaxis quadrant badge injected by overrides/main.html — producing a stray duplicate badge. Solution: remove all \`tags:\` front matter from doc pages AND remove the tags plugin from mkdocs.yml entirely. Affected files were explanation/index.md, explanation/role-of-recurrence.md, explanation/why-snakemake.md, user-guide/installation.md.

<!-- lore:019f142a-ee70-75a4-9ffb-a43516bac3d9 -->
* **DynVision docs: Python-Markdown requires blank line before bullet lists following a paragraph**: Trap: bullet lists immediately after paragraph text look valid in most Markdown renderers. Fix: Python-Markdown (used by MkDocs) requires a blank line between a paragraph and a following list or the list renders as flat continuation text. In reference/index.md this caused subsection lists (Dynamics Solvers, Data Processing, Visualization, etc.) to render as flat paragraphs instead of \`\<ul>\` elements. Always add a blank line before any bullet list that follows paragraph text.

<!-- lore:019f142a-ee7a-7cd1-9615-2cea17695ca1 -->
* **DynVision docs: snakemake dependency requires Python >=3.11, incompatible with Python 3.10 envs**: Trap: \`pip install dynvision\[doc]\` looks like the correct way to get doc dependencies. Fix: dynvision==0.1.0 depends on \`snakemake~=9.1.5\` which requires Python >=3.11; installing in a Python 3.10 environment fails with 'no matching distribution'. Workaround: activate the dedicated DynVision conda environment (Python >=3.11) rather than installing into an arbitrary env, or install doc dependencies directly without going through the package.

<!-- lore:019f050e-f793-7fb0-a5ad-d6dc5e151305 -->
* **Release tag must point to a commit that includes CI/CD workflows**: Trap: creating a tag on an early commit looks correct if the tag name matches the intended release version. Fix: verify the tagged commit contains .github/workflows/ and .zenodo.json before pushing. In DynVision, tag v0.1 initially pointed to commit 4414f1f (predating all workflow additions by ~60 commits), so PyPI publish and Zenodo archival never triggered. Recreated on dd24be4 (upstream/main HEAD) and force-pushed to fix.

<!-- lore:019f050e-f79f-7eaa-ad66-8b61122913a4 -->
* **Zenodo archival requires a GitHub Release event, not just a tag push**: Trap: pushing a tag looks equivalent to a release because GitHub shows it under 'Tags', and Zenodo docs mention tags. Fix: Zenodo's GitHub webhook fires only on a GitHub Release event (not bare tag pushes). Also requires the repo to be explicitly toggled on at zenodo.org/account/settings/github/. For DynVision, the release must be created on rgutzen/DynVision (fork) because Zenodo account lacks access to Lindsay-Lab/DynVision.

### Pattern

<!-- lore:019f050e-f7ad-7607-9cd9-631bd9a1e9ed -->
* **DynVision release procedure**: Steps: 1. Ensure all CI/CD workflows and .zenodo.json are present on the target commit — tag must not predate these files. 2. Merge dev branch PR from rgutzen/DynVision to Lindsay-Lab/DynVision upstream/main. 3. Switch to main branch on local clone, then sync origin (rgutzen/DynVision) fork to upstream/main. 4. Push updated main to origin. 5. Create and push tag (e.g. v0.1) to origin. 6. Create a GitHub Release on rgutzen/DynVision (not Lindsay-Lab) to trigger Zenodo webhook. 7. Verify Zenodo deposit at zenodo.org/uploads and replace README.md Zenodo DOI placeholder (XXXXXXXX) with assigned DOI. Gotchas: - Tagging an old commit: workflows absent, PyPI/Zenodo never trigger. Always verify .github/workflows/ exists on tagged commit. - Pushing tag without GitHub Release: Zenodo webhook does not fire on bare tag pushes. - Using upstream repo for Zenodo: Zenodo account only has access to rgutzen/DynVision fork. Verify: - \[ ] zenodo.org/account/settings/github/ shows rgutzen/DynVision toggled on. - \[ ] zenodo.org/uploads shows new deposit with DOI. - \[ ] PyPI shows new dynvision version published. - \[ ] README.md Zenodo badge updated with real DOI.

### Preference

<!-- lore:019f0517-b5d4-750f-a26c-e3aad55cc0c4 -->
* **Prefer structure-first, content-later approach when redesigning documentation**: When redesigning DynVision documentation, Robin Gutzen scopes each pass to structure + theme + placeholders only — deferring prose writing explicitly. Conventions: keep existing directory names on disk (user-guide/, tutorial/, etc.) and relabel in nav only to avoid URL breakage. Developer Guide gets a separate nav tab; License/Contributing go to footer. Task lists use explicit priority levels and completion tracking. Flag broken links and config issues immediately; create placeholder pages rather than writing full content; confirm scope boundaries before drafting prose.
